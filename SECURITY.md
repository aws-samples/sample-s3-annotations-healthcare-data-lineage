# Security Policy

## Disclaimer

This project is provided as sample/educational code and is NOT intended for production use without
additional security hardening. See the "Production Hardening Recommendations" section below.

## Reporting Vulnerabilities

If you discover a security vulnerability in this project, please report it by emailing
aws-security@amazon.com or via our [vulnerability reporting page](https://aws.amazon.com/security/vulnerability-reporting/). Do not report security vulnerabilities through public GitHub issues.

## AWS Services Used

- Amazon S3 — stores healthcare data across Bronze/Silver/Gold layers with S3 Annotations for lineage metadata
- AWS Lambda — processes S3 PutObject events and writes ingestion lineage + consent annotations
- AWS Glue — runs ETL jobs transforming data between layers (HL7→FHIR, Patient 360 aggregation)
- Amazon Athena — queries S3 Metadata Annotation Tables for lineage and compliance audits
- Amazon EventBridge — routes S3 object creation events to Lambda
- AWS CloudTrail — audits all S3 data events with log file validation
- Amazon CloudWatch — monitors Lambda invocations and detects anomalous activity
- AWS CloudFormation — infrastructure as code for all resources

## Prerequisites and Permissions

To deploy this solution, you need:

- An AWS account with permissions to create S3 buckets, Lambda functions, Glue jobs, EventBridge rules, CloudTrail trails, IAM roles, Athena workgroups, and CloudWatch alarms
- AWS CLI configured with credentials
- Python 3.13+
- Bash shell

## Known Security Considerations

| Item | Category | Rationale |
|------|----------|-----------|
| IAM roles grant s3:GetObject on scoped buckets | Scanner False Positive | KICS flags any s3:GetObject as "data exfiltration" regardless of resource scope; all roles are restricted to specific bucket ARNs — not wildcards |
| CloudTrail single-region only | Security Debt | Demo is single-region; production should enable multi-region |
| CloudTrail logs use SSE-S3 (not KMS) | Security Debt | KMS adds cost; SSE-S3 provides encryption at rest |
| CloudWatch Logs not KMS-encrypted | Security Debt | Lambda logs don't contain PHI (redaction implemented) |
| AWSGlueServiceRole managed policy | Security Debt | Custom policy preferred for production |
| Lambda not in VPC | Security Debt | Lambda only accesses S3 public endpoint; VPC adds latency |
| No Dead Letter Queue on Lambda | Security Debt | Adds complexity; implement for production reliability |
| Synthetic data only | By Design | All patient data is generated by Faker library — no real PHI/PII |

## Production Hardening Recommendations

Before using this code in a production environment, implement the following changes:

### Encryption & Transport
- **KMS Encryption**: Replace SSE-S3 with customer-managed KMS keys for PHI buckets
- **CloudTrail**: Enable multi-region trail and KMS encryption for log files
- **CloudWatch Logs**: Enable KMS encryption on log groups

### Access Control
- **VPC**: Deploy Lambda in a VPC with VPC endpoints for S3, Glue, and STS
- **Source IP Restriction**: Add S3 bucket policies restricting access to specific VPC endpoints or CIDR ranges
- **MFA**: Enable MFA on IAM principals with write access to data buckets
- **SCPs**: Use AWS Organizations SCPs to prevent accidental public access
- **IAM**: Replace AWSGlueServiceRole with a custom least-privilege managed policy

### Monitoring & Reliability
- **S3 Access Logging**: Already enabled on all data buckets via CloudFormation
- **Alarm Actions**: Configure SNS topics for CloudWatch Alarm notifications
- **DLQ**: Add Dead Letter Queue to Lambda for failed event processing
- **Input Validation**: Add pre-processing Lambda for Bronze uploads (file size, format checks)
- **AWS Config**: Configure rules for drift detection

### Data Protection
- **Object Lock**: Enable S3 Object Lock (GOVERNANCE or COMPLIANCE mode) on data buckets to prevent annotation tampering
- **Lifecycle Policies**: Move data to Glacier after retention period
- **Deletion Protection**: Add DeletionPolicy: Retain on stateful CloudFormation resources
- **AWS Backup**: Enable automated backup schedules for S3 data

## Compliance

This sample demonstrates patterns for HIPAA, GDPR, and SOX compliance workflows. It is **not** certified or validated for any compliance framework. Organizations must perform their own compliance assessments before production deployment.

## Resource Cleanup

To remove all resources deployed by this project:

1. Run the teardown script: `cd infrastructure && ./teardown.sh`
2. The script will empty all S3 buckets and delete the CloudFormation stack
3. Verify no orphaned resources remain in the AWS Console
